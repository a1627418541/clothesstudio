## Diagnosis and recommendation flow

- `src/app/api/diagnosis/route.ts` creates the diagnosis/photos transaction first, then performs one diagnosis AI call through `StyleAiService`.
- The current route passes AI recommendations into `buildMatchedRecommendations()`, so recommendation copy remains AI-authored and is only bound to an archetype afterward.
- Diagnosis analysis fields and all three recommendations are persisted together in a Prisma transaction.
- Persistence failure finalizes the diagnosis AiJob as `PERSISTENCE_FAILED`; it is not currently rewritten as an eligibility fallback.
- `src/lib/ai/style-ai-service.ts` creates only a `DIAGNOSIS_ANALYSIS` AiJob. No style-preview attempt currently creates a `STYLE_GENERATION` AiJob.

## Style-preview flow

- `src/app/api/diagnosis/[id]/style-previews/route.ts` loads the live `StyleArchetype` relation, including its `imagePromptTemplate` and DNA fields.
- A candidate is claimed by changing its observed current status to `PROCESSING`. Initial generation and failed retry are not represented as separate explicit claim commands yet.
- `src/lib/ai/style-preview-service.ts` builds an archetype template prompt when a live relation is present; otherwise it builds the generic legacy prompt.
- The service calls the provider, stores the result in R2, and only then returns the final prompt to the route.
- The route persists `previewImagePrompt` together with `COMPLETED` or `FAILED`, after provider execution. Sprint 3.8 must persist the exact compiled prompt before the paid call.

## Public provider contract

`src/lib/ai/style-preview-image-provider.ts` accepts only:

- `prompt`
- optional supported `size`

It returns only:

- nullable `url`
- optional nullable `base64`
- optional nullable `error`

EvoLink task IDs or provider result IDs are not exposed by this public interface. Sprint 3.8 must not change that protocol. Attempt number, correlation ID, stable error code, and any safely available external recovery identifier belong in the internal AiJob audit payload or server logs.

## Implementation constraints established by the audit

1. Reuse `PERSISTENCE_FAILED`, existing `ImageStatus`, `STYLE_GENERATION`, and `colorPalette`.
2. Keep the provider public protocol unchanged and inject mock providers in all tests.
3. Separate true legacy prompt generation from validated Archetype V2 snapshot compilation.
4. Do not read the live archetype relation for V2 report or prompt content.
5. Persist the compiled prompt and compiler version before a paid provider call.
6. Require exact-state CAS claims: `PENDING` for first generation and `FAILED` for explicit retry.
7. Never translate infrastructure persistence failures into legacy domain fallback.
8. Do not migrate, recompute, or overwrite old report data or old `previewImagePrompt` values.
