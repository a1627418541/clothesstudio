# Sprint 3.8 Style Archetype Calibration V2：生产发布与回滚 Runbook

## 1. 目标与硬性边界

本 Runbook 用于将 Sprint 3.8 发布到 Vercel + Neon。发布必须始终满足：

- `STYLE_ARCHETYPE_V2_ENABLED` 只有精确值 `true` 才启用 V2；缺失、拼写错误或其他值都保持 legacy。
- migration 只增加 enum、列和索引，不删除、不重命名旧字段，不回填或重算旧报告。
- 旧记录继续保持 `sourceMode=LEGACY_AI`，`archetypeSnapshot`、`archetypeVersion`、`promptCompilerVersion` 为 `null`。
- seed 失败或 readiness 失败时，V2 必须保持关闭，legacy 创建与旧报告读取继续可用。
- 验证和 smoke test 使用 mock provider；未经明确批准，不发送真实图片生成请求。
- 不在命令输出、文档、截图或工单中记录 `DATABASE_URL`、API key、R2 secret 等敏感值。

## 2. 发布角色与停止条件

建议至少明确一名发布执行人和一名复核人。以下任一条件出现时立即停止，不进入下一步：

- release commit 与已验证 commit 不一致；
- `prisma validate`、测试、类型检查或构建失败；
- migration 出现 destructive SQL、执行失败或目标数据库不正确；
- seed 未完整提交、readiness `ready` 不为 `true`；
- Vercel Production 环境中 feature flag 无法确认仍为 `false`；
- legacy 创建或旧报告读取 smoke test 失败。

基础设施失败不能伪装成 legacy domain fallback。Neon/Prisma transaction、持久化或 Vercel 部署失败时，应保留错误状态和 correlation ID，禁止通过刷新页面自动触发付费重试。

## 3. 发布前离线门禁

在准备发布的同一 commit 上执行。PowerShell 示例：

```powershell
$env:STYLE_ARCHETYPE_V2_ENABLED='false'
$env:AI_PROVIDER='mock'
$env:STYLE_PREVIEW_PROVIDER='mock'

npx prisma validate
npx prisma generate
npm run lint
npx tsc --noEmit
npm test
npm run build
```

验收要求：

- 所有命令退出码为 `0`；
- Sprint 3.8 acceptance test 包含在 `npm test` 中；
- 测试日志没有真实 provider 网络请求；
- build 可以提示未配置本地 `DATABASE_URL`，但不能出现编译、类型或页面生成失败；
- `git status --short` 为空，或只包含发布人员已明确确认的文件。

记录 release commit：

```powershell
git rev-parse HEAD
```

只记录 commit hash，不记录环境变量值。

## 4. 确认 additive migration 边界

复核：

`prisma/migrations/20260714090000_sprint3_8_archetype_v2/migration.sql`

允许的变更只有：

- 新增 `RecommendationSource`、`MacroCategory` enum；
- 为 `StyleArchetype` 增加 V2 结构化字段；
- 为 `StyleRecommendation` 增加 source/snapshot/compiler/attempt 字段；
- 新增查询索引。

禁止出现：

- `DROP TABLE`、`DROP COLUMN`、旧字段 rename；
- 对历史 recommendation 执行 `UPDATE`；
- 重写 `previewImagePrompt`、`colorPalette` 或 `archetypeId`；
- 根据当前 StyleArchetype 回填历史 snapshot。

先在 staging/preview Neon 分支执行 migration 和旧版本应用兼容检查。确认旧代码可以容忍新增列后，才操作生产数据库。

## 5. 生产部署顺序

### 5.1 保持 V2 flag 关闭

在 Vercel Project Settings → Environment Variables 中，为 Production 明确配置：

```text
STYLE_ARCHETYPE_V2_ENABLED=false
```

不要依赖变量缺失作为长期配置。确认 Production、Preview、Development 的作用域没有误设为 `true`。

### 5.2 应用生产 migration

在受信任的发布终端设置正确的 Neon Production `DATABASE_URL`，但不要打印它。然后执行：

```powershell
npx prisma migrate status
npx prisma migrate deploy
npx prisma migrate status
```

验收：`20260714090000_sprint3_8_archetype_v2` 已应用，且没有待执行或失败 migration。

如果 migration 失败：

1. 保持 V2 flag 为 `false`；
2. 停止 seed 和新代码启用；
3. 保存 Prisma/Neon 错误代码与安全 correlation 信息；
4. 修复基础设施或 migration 后再重新评估；
5. 不自动切换为 V2，也不重算旧报告。

### 5.3 部署 V2-capable 代码，但仍关闭 flag

将已验证 release commit 部署到 Vercel Production。部署时保持：

```text
STYLE_ARCHETYPE_V2_ENABLED=false
```

确认部署 SHA 与第 3 节记录的 commit 一致。此阶段新诊断必须整体写入 `LEGACY_AI`。

### 5.4 先内存校验，再事务 seed

先执行无数据库写入的完整 manifest 校验：

```powershell
npm run seed -- --validate-only
```

预期：

```text
Validated 20 Archetype V2 rows; no database writes.
```

只有 validate-only 成功后才执行事务 seed：

```powershell
npm run seed
```

预期：

```text
Validated and seeded 20 Archetype V2 rows.
```

seed 在一个 Prisma transaction 中 upsert 20 条记录；transaction 完成后还会执行只读 readiness verification。

如果 seed 失败或 transaction 回滚：保持 flag 为 `false`。不要逐条手工补写，不要启用 V2，不要修改历史报告。

### 5.5 独立执行只读 readiness

```powershell
npm run readiness:archetype-v2
```

必须同时满足：

- `ready: true`；
- `expectedArchetypeCount: 20`；
- `eligibleArchetypeCount: 20`；
- `missingExpectedSlugs: []`；
- `invalidArchetypes: []`；
- MALE：至少 3 个 eligible candidates、至少 3 个 macroCategory；
- FEMALE：至少 3 个 eligible candidates、至少 3 个 macroCategory；
- OTHER：只使用 `OTHER + UNISEX`，至少 3 个 candidates、至少 3 个 macroCategory。

当前 manifest 的预期池数量为 MALE 10、FEMALE 13、OTHER 3。数量或 macro 覆盖不符时 readiness 必须失败，V2 继续关闭。

### 5.6 Flag 关闭状态下执行 legacy smoke test

执行无付费测试：

```powershell
$env:STYLE_ARCHETYPE_V2_ENABLED='false'
$env:AI_PROVIDER='mock'
$env:STYLE_PREVIEW_PROVIDER='mock'
npx vitest run src/app/api/diagnosis/route.test.ts src/lib/diagnosis-service.test.ts
```

同时在 Production 只读打开一份 migration 之前创建的旧报告，确认：

- 页面可打开；
- 旧标题、建议、已存在图片 URL 未改变；
- 不因为读取页面而自动重调 provider；
- 无 snapshot 的记录仍按 legacy adapter 展示。

不要为了 smoke test 在 Production 触发真实图片生成。

### 5.7 开启 V2

只有第 3～5.6 节全部通过后，才在 Vercel Production 将变量改为：

```text
STYLE_ARCHETYPE_V2_ENABLED=true
```

Vercel 环境变量变更后，重新部署同一 release commit。再次确认部署 SHA 未变化。

### 5.8 执行无付费 V2 smoke test

先运行本地/CI acceptance 与路由集成门禁：

```powershell
$env:STYLE_ARCHETYPE_V2_ENABLED='true'
$env:AI_PROVIDER='mock'
$env:STYLE_PREVIEW_PROVIDER='mock'
npx vitest run src/lib/style-archetype/sprint3-8-acceptance.test.ts src/app/api/diagnosis/route.test.ts 'src/app/api/diagnosis/[id]/style-previews/route.test.ts'
```

如需端到端浏览器 smoke test，只能使用配置了 mock text/image provider 的 staging/preview 环境。验收：

- 新报告三条 recommendation 都是 `ARCHETYPE_V2`；
- 三条 snapshot 非空且 schemaVersion 为 1；
- Top 3 优先覆盖三个 macroCategory；
- title、advice、ReportDisplayModel 和 final Prompt 都来自同一 snapshot；
- Old Money、Business Formal、Streetwear、Japanese Minimal 的 anchor 约束存在；
- dry-run 不写 `previewImagePrompt`、不改变 image status、不调用 provider。

Production 中不要临时切换 provider 或用页面刷新测试失败重试。

## 6. 监控与费用保护

发布后重点观察：

- diagnosis AiJob 的 `recommendationPipeline` diagnostics；
- `fallbackReason` 是否异常集中为 `INSUFFICIENT_ELIGIBLE_ARCHETYPES` 或 snapshot 校验失败；
- style preview attempt 的 correlation ID、attempt number、expected status；
- `PERSISTENCE_FAILED` 与 `RESULT_PERSISTENCE_FAILED`；
- provider 成功但 URL 持久化失败的恢复标识和安全日志。

首次生成只能使用 `expectedStatus=PENDING`；显式失败重试只能使用 `expectedStatus=FAILED`。禁止使用 `status in [PENDING, FAILED]`，禁止页面刷新自动重试，禁止在 provider 成功但持久化失败后自动再次付费。

`StyleRecommendation` 只保存最新一次 attempt；完整历史依赖 AiJob 和 server logs。需要更长历史时另行设计 `StylePreviewAttempt` 表，不在本次发布中临时扩展。

## 7. 回滚流程

### 7.1 首要动作：关闭 V2

在 Vercel Production 设置：

```text
STYLE_ARCHETYPE_V2_ENABLED=false
```

重新部署最后一个已知稳定 commit。关闭 flag 后，新诊断稳定走 legacy；已生成的 V2 报告仍从不可变 snapshot 读取。

### 7.2 回滚时禁止的操作

- 不回滚 additive database columns/enums/indexes；
- 不删除已经保存的 `archetypeSnapshot`；
- 不把 V2 recommendation 改写为 legacy；
- 不重算历史 recommendation；
- 不覆盖历史 `previewImagePrompt`；
- 不因回滚或刷新自动重试付费 provider。

### 7.3 按失败类型处置

| 失败类型 | 处置 |
| --- | --- |
| migration 失败 | flag 保持关闭，停止 seed/deploy enable，修复 migration/基础设施 |
| migration 成功、seed 失败 | 保留 additive schema，flag 保持关闭，legacy 继续可用 |
| readiness 失败 | flag 保持关闭，修复 manifest/seed 后重新 seed 和 readiness |
| V2 selection/domain 校验失败 | 单个新诊断允许确定性整份 legacy fallback，并记录稳定 fallbackReason |
| Neon/Prisma transaction 失败 | 回滚事务，显式 PERSISTENCE_FAILED，不允许伪装 legacy fallback |
| provider 失败 | 状态 FAILED，只有用户显式操作才允许 exact FAILED retry |
| provider 成功、URL 持久化失败 | PERSISTENCE_FAILED，保存恢复标识，禁止自动付费重试 |
| 新代码运行异常 | 关闭 flag，部署已知稳定代码；保留 schema 和历史 snapshots |

### 7.4 重新启用条件

完成修复后重新执行：

```powershell
npm run seed -- --validate-only
npm run seed
npm run readiness:archetype-v2
$env:STYLE_PREVIEW_PROVIDER='mock'
npx vitest run src/lib/style-archetype/sprint3-8-acceptance.test.ts
```

全部通过后才可以重新将 Production flag 设置为 `true`。
