# 国内真实换装供应商盲测设计

## 目标

在不影响当前线上 mock 流程的前提下，用同一批上衣、下装人物图和服装图真实调用腾讯混元 `ChangeClothes` 与火山引擎 `DressingDiffusionV2`，并用腾讯单独验证连衣裙，保存结果和运行指标，为生产供应商选择提供证据。

## 本轮范围

- 支持上衣、下装和连衣裙；帽子不参与本轮测试。
- 每个测试样本包含一张单人人物图、一张正面服装图和明确的服装类别。
- 两家供应商对上衣、下装使用完全相同的输入样本。腾讯接口明确支持连衣裙；火山公开参数仅明确展示 `upper`、`bottom`，因此连衣裙在火山侧记录为不支持，不猜测未公开枚举。
- 输出包含生成图片、成功状态、耗时、供应商请求 ID 和经过脱敏的错误码。
- 云密钥仅从环境变量读取，不进入数据库、日志、测试快照或 Git。
- 当前结果页继续使用 mock provider；完成盲测并确认供应商后再切换生产流量。

## 方案选择

采用独立的“供应商适配器 + 本地盲测命令”方案，而不是立即把真实接口接入结果页。这样可以先比较身份保持、商品还原、成功率和延迟，不会让真实用户承担尚未验证的生成质量与云费用。

腾讯适配器使用官方 Node.js AIArt SDK 调用 `ChangeClothes`。火山适配器调用图片换装 V2 的提交与查询接口，并将异步任务统一为 `Promise<{ imageUrl, requestId }>`。两者都实现同一个窄接口，之后选中的实现可以接到现有 `VirtualTryOnProvider`。

## 数据流

1. 测试清单提供 `caseId`、人物图 URL、服装图 URL和 `TOP | BOTTOM | DRESS`。
2. 命令校验 URL、类别、重复 ID 和密钥配置。
3. 对每个样本依次调用支持该类别的供应商，避免默认并发 1 的账户限额。
4. 供应商返回结果后立即下载图片，避免临时 URL 失效。
5. 在 `artifacts/try-on-benchmark/<runId>/` 保存图片和 `results.json`。
6. 人工以匿名编号对图片评分：身份保持、商品还原、肢体结构、整体真实度，各 1～5 分。

## 错误和隐私

- 日志只记录供应商、样本 ID、耗时、状态和标准化错误码。
- SDK 原始异常不得直接显示到页面或写入结果文件。
- HTTP 429、服务繁忙和可重试 5xx 使用有限次数退避；参数、内容审核和鉴权失败不重试。
- 人物照片必须已有用户试穿授权；测试产物保存在被 Git 忽略的本地目录，评测完成后手动删除。
- 腾讯返回的一小时临时 URL 必须在命令运行期间下载。

## 验收标准

- 无密钥时，命令在发起网络请求前给出明确的缺失变量清单。
- 单元测试不访问外网，能够覆盖类别映射、请求参数、轮询、超时和错误脱敏。
- 同一清单可分别运行 `tencent`、`volcengine` 或 `all`。
- 至少完成 8 个代表性上衣/下装样本的双供应商真实调用和 2 个腾讯连衣裙样本，再决定生产默认值。
- 生产 provider 在盲测完成前仍为 `mock`。
