# 淘宝 / 京东真实商品推荐与本人虚拟试穿设计

日期：2026-07-20
状态：已确认，待用户审阅书面规格
适用项目：AI Personal Style Studio

## 1. 背景

当前结果页能够生成风格预览图，也已经具备结构化单品文字建议、人脸试穿授权、预览图字段和实验性的换脸 Provider。当前主链路仍是“先生成匿名模特，再把用户正脸换进去”，推荐单品也不对应真实可购买 SKU。

本次改造要把结果页升级为真实购买场景：系统从淘宝或京东选择实际商品，以用户上传的全身照为人物底图生成本人试穿图，并在结果页展示与试穿图一一对应的商品和购买入口。

## 2. 目标

1. 根据用户诊断和预算，从淘宝或京东自动组成三套可购买搭配。
2. 每套搭配只使用一个平台的商品，不混合淘宝和京东。
3. 主方案在用户授权后自动生成本人试穿图。
4. 另外两套先展示商品组合，用户点击后再生成本人试穿图。
5. 试穿图尽量忠实呈现具体 SKU 的颜色、轮廓、图案和帽型。
6. 结果图保留用户的脸、身材和原始姿势。
7. 低质量结果不冒充真实试穿；失败时安全降级为商品组合和匿名示意图。
8. 第一阶段使用模拟淘宝 / 京东商品接口和模拟生成服务跑通全流程，后续替换真实适配器时不改业务层。

## 3. 非目标

- 首版不处理鞋、包、首饰和眼镜的画面试穿。
- 首版不提供购物车聚合、代下单、支付或订单同步。
- 首版不抓取淘宝、京东网页；真实阶段使用正规联盟或开放平台接口。
- 首版不保证像素级复刻商品详情图，验收以可定义、可检测的商品一致性为准。
- 首版不训练自有虚拟试穿模型。
- 本规格不包含结果页整体品牌重做；只确定已选中的视觉方向和与本功能相关的状态展示。

## 4. 已确认的产品决策

### 4.1 商品来源

- 目标来源：淘宝联盟与京东联盟。
- 第一阶段：使用模拟适配器和内置模拟商品集。
- 后续阶段：取得联盟账号和 API 权限后接入真实搜索、商品详情和推广链接。
- 真实商品字段必须带抓取时间；价格、库存和可售状态均视为时效性快照。

### 4.2 预算

诊断表单增加整套预算档位：

- 500 元以内
- 500–1000 元
- 1000–2000 元
- 2000 元以上

预算约束作用于整套商品总价，而不是单件商品。匹配器可以保留少量价格余量，但不能静默超出用户选择的上限；无法组成完整搭配时应明确提示并给出重新选择预算的入口。

### 4.3 搭配范围

首版商品品类：

- 上衣
- 裤子或裙子
- 可选外套
- 帽子

每套搭配必须有上衣、下装和帽子，且三类商品都参与试穿合成。外套根据风格、季节与预算决定是否加入。

### 4.4 平台规则

- 每套搭配只能来自淘宝或京东中的一个平台。
- 三套方案之间可以选择不同平台。
- 选品器按风格匹配度、商品质量、预算完整性和可用图片质量决定该套使用的平台。
- 结果页的整套购买入口必须明确标识平台。

### 4.5 生成策略

- 主推荐：诊断完成并取得授权后自动生成。
- 备选 2 和备选 3：先显示商品组合、总价和平台；用户点击后按需生成。
- 重新生成默认锁定原 SKU，避免图片与商品卡片发生静默变化。
- 原 SKU 下架时，先要求用户确认替代搭配，再基于新快照生成。

## 5. 用户流程

### 5.1 已授权本人试穿

1. 用户上传正脸、侧脸和全身照。
2. 用户勾选本人试穿授权并选择整套预算。
3. 系统完成风格诊断。
4. 商品匹配器生成三套单平台搭配并保存商品快照。
5. 系统自动排队生成主方案本人试穿图。
6. 结果页立即可访问，并展示选品结果和生成进度。
7. 主图通过质量检测后自动替换进度状态。
8. 用户可查看平台商品清单，或按需生成另外两套。

### 5.2 未授权本人试穿

1. 用户仍可完成诊断和选品。
2. 结果页展示商品组合和匿名模特示意图。
3. 系统不得向任何试穿或人脸服务发送用户照片。
4. 用户可在结果页补充授权，授权后触发主方案本人试穿。

### 5.3 撤回授权

- 撤回授权后停止所有尚未开始的本人试穿任务。
- 已运行任务在进入下一个外部 Provider 步骤前再次检查授权，检查失败则取消。
- 撤回授权不删除文字报告和商品推荐。
- 用户可同时选择立即删除已经生成的人脸相关图片。

## 6. 系统架构

### 6.1 商品接入层

定义统一的 `MarketplaceProductProvider`：

```ts
interface MarketplaceProductProvider {
  platform: "TAOBAO" | "JD";
  search(input: ProductSearchInput): Promise<ProductSearchResult>;
  refresh(input: ProductRefreshInput): Promise<ProductSnapshot>;
  buildPurchaseLink(input: PurchaseLinkInput): Promise<string>;
}
```

实现：

- `MockTaobaoProductProvider`
- `MockJdProductProvider`
- 后续 `TaobaoAffiliateProductProvider`
- 后续 `JdAffiliateProductProvider`

业务层只依赖统一接口，不依赖联盟接口的原始返回结构。

### 6.2 智能选品层

`OutfitProductMatcher` 接收：

- 性别、年龄和风格原型
- 体型、脸型和风格关键词
- 推荐颜色与避雷信息
- 预算档位
- 所需品类

输出三套 `OutfitProductPlan`。每套包含平台、商品快照、总价、匹配解释和完整性评分。

候选商品按以下顺序排序：

1. 品类与风格匹配
2. 颜色和版型匹配
3. 商品主图是否适合虚拟试穿
4. 整套预算完整性
5. 商品可售状态和信息新鲜度
6. 店铺与商品质量信号

如果同一平台无法组成合格整套，不拼入另一平台商品。该平台方案应失败并尝试另一个平台。

### 6.3 分层虚拟试穿层

现有“匿名模特生成 + 换脸”不再作为本人试穿主链路。新的 Provider 边界为：

```ts
interface VirtualTryOnProvider {
  applyGarment(input: GarmentTryOnInput): Promise<TryOnStageResult>;
  applyHat(input: HatTryOnInput): Promise<TryOnStageResult>;
}

interface IdentityRestoreProvider {
  restore(input: IdentityRestoreInput): Promise<IdentityRestoreResult>;
}

interface TryOnQualityProvider {
  evaluate(input: TryOnQualityInput): Promise<TryOnQualityResult>;
}
```

合成顺序：

1. 用户全身照作为人物底图。
2. 上衣或连体主体服装。
3. 下装。
4. 外套。
5. 帽子。
6. 身份检测；必要时使用正脸照修复。
7. 最终商品一致性与图片完整性检测。

每一步把原始脸部区域、人体比例和姿势作为保护条件。中间图使用临时对象键，最终成功或失败后进入清理流程。

### 6.4 编排层

`TryOnOrchestrator` 负责状态机、授权复查、重试和清理。开发和测试环境可使用同步模拟执行器；生产使用可持久化后台任务，避免页面请求承担长时间图片任务。

状态：

- `PENDING_PRODUCTS`
- `QUEUED`
- `APPLYING_GARMENTS`
- `APPLYING_HAT`
- `RESTORING_IDENTITY`
- `QUALITY_CHECKING`
- `COMPLETED`
- `FAILED`
- `CANCELLED`
- `EXPIRED`

结果页只需映射为用户可理解的状态，不暴露 Provider 名称或内部错误。

## 7. 数据设计

### 7.1 诊断

在 `StyleDiagnosis` 中保存：

- `budgetTier`
- 已有 `faceTryOnConsent`
- 已有 `faceTryOnConsentAt`
- 可选 `faceTryOnRevokedAt`

### 7.2 商品快照

新增关系型 `RecommendationProduct`，不继续把真实 SKU 仅存入 `StyleRecommendation.items` JSON：

- `id`
- `recommendationId`
- `platform`
- `externalProductId`
- `externalSkuId`
- `category`
- `title`
- `imageUrl`
- `purchaseUrl`
- `priceCents`
- `currency`
- `sellerName`
- `color`
- `variantLabel`
- `isOptional`
- `availabilityStatus`
- `snapshotAt`
- `position`
- `rawSnapshot`，仅保存排障所需且不含密钥的数据

`StyleRecommendation` 增加：

- `marketplacePlatform`
- `productTotalCents`
- `productPlanStatus`

商品快照用于保证历史报告、试穿生成和购买卡片引用同一组 SKU。实时刷新不得静默覆盖生成时使用的快照。

### 7.3 本人试穿结果

沿用 `StyleRecommendation.tryOnImageUrl` 和 `tryOnImageStatus`，增加：

- `tryOnAttemptCount`
- `tryOnFailureCode`
- `tryOnProvider`
- `identityScore`
- `productFidelityScore`
- `tryOnExpiresAt`
- `tryOnProductSnapshotHash`

哈希由排序后的商品平台、商品 ID、SKU、颜色和版本信息生成，用于检测图片和购买卡片是否仍对应同一套商品。

## 8. 质量门槛

第一阶段模拟 Provider 固定返回可预测分数，用于验证状态机。接入真实 Provider 后，质量检测至少覆盖：

- 人脸身份相似度
- 人脸完整性与明显畸变
- 人体完整性、手脚和姿势稳定性
- 上衣、下装、外套和帽子是否出现
- 主色是否匹配 SKU
- 显著图案和帽型是否匹配
- 是否出现额外人物、文字或水印

具体阈值必须通过真实样本校准，不在没有数据的情况下写死产品承诺。系统决策只有三种：

- 通过并展示
- 自动重试一次
- 失败并降级

不得展示未通过质量门槛的本人试穿图。

## 9. 失败与降级

### 9.1 商品失败

- 搜索无结果：尝试同平台放宽非核心颜色条件。
- 仍无法组成完整搭配：尝试另一平台生成整套。
- 两个平台均失败：保留风格报告，提示调整预算或稍后重试。

### 9.2 生成失败

- 单阶段 Provider 失败：对该阶段自动重试一次。
- 身份或商品一致性不合格：整套重新生成一次。
- 最终失败：不展示错误图片，展示商品卡片和匿名示意图。
- 用户可手动重新生成，但应有频率和成本限制。

### 9.3 商品失效

- 打开购买清单前刷新可售状态。
- 商品下架时明确标记，不自动替换已经展示的试穿图对应商品。
- 用户确认替换商品后创建新的商品快照，并将旧试穿图标记为过期，再生成新版。

## 10. 结果页设计

视觉方向采用已确认的 A：时尚杂志电商。

### 10.1 主方案

- 大幅本人试穿图作为视觉中心。
- 清晰显示 AI 试穿披露和质量检查状态。
- 右侧或相邻区域展示四件真实商品卡片。
- 显示平台、整套参考价和购买清单入口。
- 商品卡片包含真实主图、名称、规格、价格和单品链接。

### 10.2 备选方案

- 显示商品组合、平台、总价和风格说明。
- 不预生成本人试穿图。
- 使用“生成这套试穿”按钮触发按需任务。

### 10.3 页面状态文案

- 正在选品
- 正在生成本人试穿
- 正在检查效果
- 效果不合格，正在重试
- 本人试穿暂不可用
- 图片已过期，请重新上传后生成

页面不得向用户展示原始 Provider 错误、密钥状态或内部堆栈。

## 11. 隐私、安全与保存周期

- 未授权时不得把用户照片发送给本人试穿、人脸修复或质量检测服务。
- 每个外部调用前重新读取授权状态，不能仅依赖任务创建时的快照。
- API Key 只存在服务端环境变量中。
- 对外部 Provider 使用短时可访问的图片 URL；不暴露永久私有对象地址。
- 匿名用户的人脸原图、全身照和本人试穿图保存 30 天，之后自动删除或标记删除并进入对象清理。
- 登录用户素材保留到用户主动删除。
- 中间合成图不长期保存，任务完成后及时清理。
- 日志不得记录人脸图片内容、签名 URL、API Key 或完整联盟推广凭证。

## 12. API 边界

第一阶段需要的业务端点：

- `POST /api/diagnosis`：接收预算与授权，创建诊断。
- `GET /api/diagnosis/:id`：返回诊断、商品快照和本人试穿状态。
- `POST /api/diagnosis/:id/recommendations/:recommendationId/try-on`：为备选或失败方案按需生成。
- `POST /api/diagnosis/:id/try-on-consent`：补充或撤回授权。
- `POST /api/diagnosis/:id/recommendations/:recommendationId/refresh-products`：显式刷新失效商品。

所有端点沿用现有用户 / 匿名会话所有权校验。生成端点还必须校验授权、商品快照完整性和任务幂等性。

## 13. 可观测性

记录但不泄露敏感内容：

- 平台搜索成功率与耗时
- 完整搭配生成率
- 预算内命中率
- 主方案生成耗时
- 各生成阶段失败率
- 自动重试率
- 人脸和商品一致性通过率
- 备选方案按需生成点击率
- 淘宝 / 京东购买入口点击率
- 商品失效率

## 14. 测试策略

### 14.1 单元测试

- 预算档位与整套总价约束。
- 单套商品平台一致性。
- 商品品类完整性与可选外套规则。
- 商品快照哈希稳定性。
- 授权检查和撤回逻辑。
- 状态机转换、幂等和单次自动重试。
- 匿名素材 30 天过期计算。

### 14.2 集成测试

- 模拟淘宝成功、京东失败及反向场景。
- 两个平台均无法组成整套时安全降级。
- 已授权用户自动排队主方案。
- 未授权用户不调用任何人脸相关 Provider。
- 备选方案仅在点击后生成。
- 商品下架后不静默替换。
- 质量不合格时重试一次后降级。

### 14.3 页面测试

- 选品、生成、检查、完成、失败和过期状态均可正确渲染。
- 主图与商品快照哈希一致时才展示。
- 单品和整套购买入口平台标识清楚。
- 移动端商品卡片和备选方案可阅读、可操作。
- AI 试穿披露和价格时效提示始终可见。

### 14.4 隐私回归测试

- 未授权请求中不存在正脸、侧脸或全身图 URL。
- 撤回授权后待执行任务被取消。
- 日志和错误响应不包含签名 URL 或密钥。
- 匿名素材到期后数据库记录和对象存储均进入清理。

## 15. 分阶段交付

### 阶段 1：模拟全流程

- 预算字段与表单。
- 真实商品快照数据结构。
- 淘宝 / 京东模拟商品适配器。
- 三套单平台选品。
- 模拟分层试穿、身份修复和质量检测。
- 主方案自动生成，备选按需生成。
- 结果页新状态与真实商品卡片。

### 阶段 2：真实图片生成

- 接入虚拟试穿 Provider。
- 接入帽子合成与身份修复 Provider。
- 用真实样本校准质量阈值。
- 实现中间图清理、重试和成本控制。

### 阶段 3：真实联盟商品

- 接入淘宝联盟与京东联盟。
- 实现真实搜索、商品刷新和推广链接。
- 实现商品下架与替换确认流程。
- 增加购买点击和商品失效监控。

## 16. 验收标准

第一阶段完成时：

1. 用户能选择整套预算并完成诊断。
2. 系统返回三套完整、单平台的模拟真实商品搭配。
3. 已授权用户的主方案自动进入本人试穿任务。
4. 未授权用户不会触发人脸相关任务。
5. 两套备选只有点击后才生成。
6. 结果页能展示所有生成状态、商品、总价和平台购买入口。
7. 质量失败时不展示错误人脸或不对应商品的试穿图。
8. 匿名素材具备 30 天过期与清理路径。
9. 自动化测试覆盖平台一致性、授权、重试、降级和过期。

第二、三阶段各自在接入真实服务后增加真实 Provider 合约测试和受控端到端试跑，不以模拟测试替代真实能力验收。
